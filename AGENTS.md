# Project guide

## Project overview

Windows-only PowerShell scripts to install prerequisites, configure, build, and run [llama.cpp](https://github.com/ggerganov/llama.cpp) with CUDA, Vulkan, and ROCm (HIP) support, and NSIS installer packaging. Not compatible with Linux or macOS. llama.cpp's built-in chat interface (served by `llama-server`) is used as the frontend â€” no separate web UI is bundled.

## Structure

- `build\config-build.psd1` â€” build-time configuration (paths, GPU targets, compiler flags). Generated by `01-configure.ps1`; lives under `build\` (whole dir gitignored). Runtime / per-model settings are **not** stored at repo root â€” they live under `%LOCALAPPDATA%\llama.cpp\config\` and are written by `resources\config-server.ps1` / `config-model.ps1` on first launch (or by the NSIS install-time page).
- `common.ps1` â€” shared bootstrap: loads `$cfg` from `build\config-build.psd1`, adds ROCm/HIP\bin to PATH, exposes `Enable-VsDevShell` as a function. VS Dev Shell activation is **opt-in**: only build/package scripts call it.
- `00-install-prerequisites.ps1` â€” idempotent toolchain bootstrapper: installs **and** updates the winget-tracked packages (PowerShell 7+ / OpenSSL / NSIS) in a single UAC-elevated session (`winget install` for missing, `winget upgrade` for present), then `git pull --ff-only`s `$cfg.LlamaCppDir` when cloned. Flags CUDA / Vulkan / AMD HIP for manual install. Reads `build\config-build.psd1` directly (does **not** dot-source `common.ps1`) so it works on a fresh machine before `01-configure.ps1` has run. End-of-run report shows beforeâ†’after versions, plus a rebuild recommendation (`02-build.ps1` / `03-package.ps1`) only when llama.cpp source moved. Safe to re-run any time.
- `01-configure.ps1` â€” auto-detects paths, activates VS Dev Shell for tool discovery, verifies tools, creates `build\` if missing, writes `build\config-build.psd1`. Accepts `-LlamaCppDir` param (defaults to `$PSScriptRoot\build\llama.cpp`). Writes GPU targets (`gfx9xx`/`gfx1x`/`gfx12xx`), build type (Release), compiler (clang), march flags (`-march=x86-64-v3`), and build jobs.
- `02-build.ps1` â€” dot-sources `common.ps1` + calls `Enable-VsDevShell`. CMake configure (out-of-source) + build with Ninja. Auto-clones/pulls llama.cpp into `build\llama.cpp\`, builds into `build\llama.cpp-cmake\`. Uses sccache if available. All under `.\build\` (gitignored).
- `03-package.ps1` â€” dot-sources `common.ps1` + calls `Enable-VsDevShell` (cmake --install needs the VS env). Stages from `build\llama.cpp-cmake\` into `build\staging\`, then builds NSIS installer to `.\dist\`.
- `resources/run-model.ps1` â€” runtime entry point shipped with the installer (main "llama.cpp" Start Menu shortcut). Launches `llama-server` in **router mode** against the user-owned `%LOCALAPPDATA%\llama.cpp\config\presets.ini` via `--models-preset`. The router lazy-loads each `[section]` on its first matching request and unloads the previous one once `--models-max` is reached. Auto-invokes `config-server.ps1` / `config-model.ps1` if their outputs are missing. Accepts `-ServerExe` (defaults to `$PSScriptRoot\bin\llama-server.exe`) as an override to point at an in-tree build (`.\build\llama.cpp-cmake\bin\llama-server.exe`) instead of the installed binary. Tees stdout+stderr to `%LOCALAPPDATA%\llama.cpp\logs\llama-server.log`. CWD is forced to `%LOCALAPPDATA%\llama.cpp` so first-run files don't land in Program Files.
- `resources/config-server.ps1` â€” llama-server config writer. Writes `%LOCALAPPDATA%\llama.cpp\config\server.ini` (a single `[Server]` section). Holds machine-wide knobs only (Port, Hostname, Mlock, Threads, CacheReuse, ThreadsBatch, **ModelsMax**, ModelsDir); per-model params live in `presets.ini`. NSIS calls it with `-NonInteractive -Port -Hostname -Threads -ThreadsBatch -ModelsDir` from the install-time page (ModelsMax defaults to 1 â€” strict one-at-a-time switching; tweakable via the interactive prompt afterwards). Re-runnable via "Configure llama-server" Start Menu shortcut (the window auto-closes once the file is written â€” no `-NoExit`). Also supports `-DumpIni <path>`: emits the current `[Server]` values to a UTF-16 LE INI and exits without modifying anything; used by NSIS `.onInit` to surface previous values as install-page defaults.
- `resources/config-model.ps1` â€” per-model preset builder. Operates directly on `%LOCALAPPDATA%\llama.cpp\config\presets.ini` â€” the file consumed by `llama-server --models-preset`. Lists `.gguf` files under `ModelsDir` (marks already-configured presets with `*`), prompts for every model param (ctx-size, n-gpu-layers, parallel, batch-size, ubatch-size, cache-type-k/v, flash-attn, cache-ram (per-model prompt-cache RAM limit in MiB â€” `--cache-ram`; default 8192, -1 = no limit, 0 = disable), jinja, reasoning `on`/`off`/`auto` (the `--reasoning` flag â€” replaces the deprecated `enable_thinking` chat-template-kwarg; a legacy `enable_thinking` kwarg in an existing section is auto-migrated to `reasoning` and stripped from `chat-template-kwargs`), reasoning-format `auto`/`none`/`deepseek`, n-cpu-moe, sampling incl. min-p, chat-template-kwargs), then **rewrites just the matching `[section]` while preserving every other section in the file byte-for-byte** (including comments and hand-added exotic keys). The section header is the id clients pass as `"model"` in API requests. Hand-edits within the section being re-wizarded are lost; hand-edits to *other* sections survive. Also refreshes `server.ini`'s `ModelsDir` pointer (in-place, comment-preserving). Re-runnable via "Configure model" Start Menu shortcut (window auto-closes on completion).
- `resources/common-functions.ps1` â€” shared helpers dot-sourced by all three runtime scripts (`config-server.ps1`, `config-model.ps1`, `run-model.ps1`). Currently exposes `Read-ServerIni` (parse the `[Server]` section into a hashtable of strings) and `Set-ServerIniField` (replace one key in `[Server]` while preserving comments, key order, and any other sections). Staged into `$INSTDIR` alongside the other runtime scripts and also extracted to `$PLUGINSDIR` in NSIS `.onInit` so the `-DumpIni` invocation there can dot-source it.
- `llama-cpp.nsi.template` â€” NSIS installer template with placeholders (`@VERSION@`, `@STAGING_DIR@`, `@OUTPUT_FILE@`) replaced by `03-package.ps1` into `build\llama-cpp.nsi` (transient â€” deleted after the build). Has one custom nsDialogs page (server config: Port / CPU threads / Batch threads / Bind / Models folder browser) that drives `config-server.ps1 -NonInteractive` post-install to write `server.ini`. Thread defaults are pre-computed in `.onInit` from `%NUMBER_OF_PROCESSORS%` using the same formulas as the runtime fallback. Models folder defaults to `$PROFILE\.llama.cpp\models` (mirrored in `config-server.ps1`'s interactive fallback) and is created on disk if missing. When an existing `server.ini` is present, `.onInit` extracts `config-server.ps1` and `common-functions.ps1` to `$PLUGINSDIR` and runs the script with `-DumpIni` to override these defaults with the user's previous choices. Model-dependent params (CtxSize, GpuLayers, Parallel, CacheTypeK/V, FlashAttn, sampling, etc.) are NOT prompted at install â€” created lazily per-model by config-model.ps1 on first run. Start Menu shortcuts target `pwsh.exe` (PowerShell 7+).

## Key conventions

- `common.ps1` is dot-sourced by all scripts that need `$cfg` or ROCm PATH (everything except `00-install-prerequisites.ps1`, `01-configure.ps1`, and the `resources/` runtime scripts). Build/package scripts additionally call `Enable-VsDevShell`.
- The Threads / ThreadsBatch defaults (`config-server.ps1`) and the runtime fallbacks (`resources\run-model.ps1`) use the **same formulas**: `Threads = max(1, floor(cores * 0.5))`, `ThreadsBatch = max(1, floor(cores * 0.75))`. Don't let them drift.
- `build\config-build.psd1` (build-time) uses single-quoted PowerShell-data-file syntax. All other generated config files (`server.ini`, `presets.ini`) are plain INI written as UTF-8 without BOM (consistent encoding so `common-functions.ps1`'s parser and llama-server's `--models-preset` reader both see the same bytes).
- All build artifacts (source clones, cmake binary dir, NSIS staging, sccache cache, `config-build.psd1`, generated `.nsi`) live under `.\build\` (gitignored). The final installer goes to `.\dist\`. `rm -rf build\` is a full clean wipe â€” it nukes the sccache too, so the next full build re-fills it from scratch.
- The installer registers in `HKLM\Software\llama.cpp` (InstallDir) and supports upgrades (uninstalls previous version first). User runtime state lives under `%LOCALAPPDATA%\llama.cpp\`: `config\server.ini` (llama-server runtime + ModelsDir + ModelsMax â€” single `[Server]` section), `config\presets.ini` (per-model presets â€” one `[section]` per model, the file consumed by `llama-server --models-preset`; hand-edits are preserved across `config-model.ps1` runs for any section the wizard does not touch), `logs\llama-server.log`. The uninstaller asks (defaulting to No) whether to delete the whole tree, so it survives reinstall/upgrade by default.
- sccache is auto-detected; cache stored in `build\.sccache\`. Uses `SCCACHE_MAX_FRAME_LENGTH=104857600` for large GPU objects.
