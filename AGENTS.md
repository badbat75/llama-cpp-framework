# Project guide

## Project overview

Windows-only PowerShell scripts to install prerequisites, configure, build, and package [llama.cpp](https://github.com/ggerganov/llama.cpp) with CUDA, Vulkan, and ROCm (HIP) support, and NSIS installer packaging. Not compatible with Linux or macOS. llama.cpp's built-in chat interface (served by `llama-server`) is used as the frontend — no separate web UI is bundled.

A companion Rust binary (`llama-cpp-config`) provides a GUI + CLI configurator for `server.ini` and per-model `presets.ini`, replacing what were previously PowerShell config scripts.

## Structure

- `build\config-build.psd1` — build-time configuration (paths, GPU targets, compiler flags). Generated by `01-configure.ps1`; lives under `build\` (whole dir gitignored).
- `common.ps1` — shared bootstrap: loads `$cfg` from `build\config-build.psd1`, adds ROCm/HIP\bin to PATH, exposes `Enable-VsDevShell` as a function. VS Dev Shell activation is **opt-in**: only build/package scripts call it. Dot-sourced by `02-build.ps1` and `03-package.ps1`.
- `00-install-prerequisites.ps1` — idempotent toolchain bootstrapper: installs **and** updates the winget-tracked packages (PowerShell 7+ / OpenSSL / NSIS) in a single UAC-elevated session (`winget install` for missing, `winget upgrade` for present), then `git pull --ff-only`s `$cfg.LlamaCppDir` when cloned. Flags CUDA / Vulkan / AMD HIP for manual install. Reads `build\config-build.psd1` directly (does **not** dot-source `common.ps1`) so it works on a fresh machine before `01-configure.ps1` has run. End-of-run report shows before→after versions, plus a rebuild recommendation (`02-build.ps1` / `03-package.ps1`) only when llama.cpp source moved. Safe to re-run any time.
- `01-configure.ps1` — auto-detects paths, activates VS Dev Shell for tool discovery, verifies tools, creates `build\` if missing, writes `build\config-build.psd1`. Accepts `-LlamaCppDir` param (defaults to `$PSScriptRoot\build\llama.cpp`). Writes GPU targets (`gfx9xx`/`gfx1x`/`gfx12xx`), build type (Release), compiler (clang), march flags (`-march=x86-64-v3`), and build jobs.
- `02-build.ps1` — dot-sources `common.ps1` + calls `Enable-VsDevShell`. Builds **both** llama.cpp (CMake configure + Ninja) **and** `llama-cpp-config` (cargo build --release). Auto-clones/pulls llama.cpp into `build\llama.cpp\`, builds CMake output into `build\llama.cpp-cmake\`, and copies the Rust binary to `build\llama-cpp-config\`. Uses sccache if available for the C++ build. All under `.\build\` (gitignored).
- `03-package.ps1` — dot-sources `common.ps1` + calls `Enable-VsDevShell` (cmake --install needs the VS env). Stages from `build\llama.cpp-cmake\` + `build\llama-cpp-config\` into `build\staging\`, then builds NSIS installer to `.\dist\`. The package is named `llama-cpp-framework-v<frameworkVersion>-<llamaBuild>-<arch>-setup.exe`, e.g. `llama-cpp-framework-v1.0.0-b3456-x64-setup.exe`: **frameworkVersion** = the `llama-cpp-config` crate version (parsed from `llama-cpp-config\Cargo.toml` `[package]`; the configurator and the framework are versioned together), **llamaBuild** = `git describe --tags` of `$cfg.LlamaCppDir` (e.g. `b3456`), **arch** = `x64`/`arm64`.
- `llama-cpp.nsi.template` — NSIS installer template with placeholders (`@VERSION@` = framework/configurator version, `@LLAMA_BUILD@` = bundled llama.cpp build, `@STAGING_DIR@`, `@OUTPUT_FILE@`) replaced by `03-package.ps1` into `build\llama-cpp.nsi` (transient — deleted after the build). The ARP DisplayName/installer title read "llama.cpp-framework v@VERSION@ (llama @LLAMA_BUILD@)"; DisplayVersion = framework version; the llama build is also written to `HKLM\Software\llama.cpp\LlamaBuild`. Three components: binaries (required), PATH addition (optional), Start Menu shortcut (optional). Installs `bin\` (llama.cpp binaries, plus `llama-cpp-config.exe` and `llama.ico` side-by-side with `llama-server.exe`) and `lib\`. **No custom config page** — runtime configuration is handled entirely by `llama-cpp-config` post-install. Start Menu shortcut: "llama.cpp Config" → `llama-cpp-config.exe`.
- `llama-cpp-config\` — Rust GUI + CLI configurator for `server.ini` + per-model `presets.ini`. This entry is the **map**; depth lives in each module's header comment and in [llama-cpp-config\README.md](llama-cpp-config\README.md) — read those before diving in, not this file.
  - **GUI shell** (`gui.rs` + `ui\app.slint`) — three tabs (**Server** → `server.ini`, **Models** → per-model `presets.ini`, **Integrations** → `opencode.json` model list + Claude Code snippet). The nav rail's bottom carries the server run controls (Start/Stop over Open-chat over Refresh), reachable from any tab; a footer shows llama-server run state + version. All shared UI state (every property + callback the pages and Rust both touch) lives in ONE `export global AppState` in `ui\state.slint`; Rust drives it via `app.global::<AppState>()`, the pages read/write `AppState.<name>` directly. `ui\app.slint` is just the `AppWindow` window chrome + tray (nav rail, run controls, modals, footer); each tab body is a separate component file — `ui\server_page.slint`, `ui\models_page.slint`, `ui\integrations_page.slint` — with shared structs in `ui\types.slint` and shared cards (`SectionCard`, `LabeledField`) in `ui\components.slint`. `gui.rs::run()` builds the window then wires each tab via `wire_server_tab` / `wire_models_tab` / `wire_integrations_tab` / `wire_tray`; form ↔ preset conversion lives in `src\form.rs`, and the hidden process-launch helper (`CREATE_NO_WINDOW`) in `src\proc.rs`. **Adding a top-level UI property is now a one-file change** (`ui\state.slint`) — no per-page re-declaration or forwarding.
  - **Server tab** — port / host / threads / models-dir, machine-wide multi-GPU split (`SplitMode`/`TensorSplit`), and a read-only **Command Line** card: `runstate::command_line()` renders the exact `llama-server` invocation one `--flag value` per line, joined with the shell continuation char (`` ` `` on Windows, `\` on POSIX) so it pastes into a terminal.
  - **Models tab** — preset editor grouped into cards: **Assets** (model / mmproj / one merged draft picker feeding `--model-draft` + spec-type), **Model info** (read-only GGUF box), **Hardware Config** (device / split / layer knobs), and **Speculative** (`spec-draft-n-max`). The three layer fields (`n-gpu-layers`, `n-cpu-moe`, `n-gpu-layers-draft`) are sliders bounded by the GGUF `block_count`, each with an "auto" checkbox that omits the flag. Many controls are guardrailed by Slint `enabled` bindings (speculative fields need a draft available; MoE CPU layers needs an MoE model; split fields disable when a single device is pinned) — exact conditions in the README.
  - **Model-info box** (`gguf.rs`) — reads GGUF metadata via llama.cpp's **own** reader (runtime-loaded `ggml-base.dll`, no reimplemented parser): dense/MoE split, layers, trained context, GQA, quant, embedded MTP, plus mmproj (clip) and draft headers. Synchronous + uncached; shows "unavailable" with no fallback if the DLL can't load. See the `gguf.rs` header.
  - **Speculative / draft caveat** — gemma4-assistant MTP heads (`n_layer=0`) crash under multi-device "auto" memory-fitting. To run the draft on GPU, pin model **and** draft to one device (`device`/`device-draft = CUDA0`) with room for the chosen context; leaving the draft on CPU (`n-gpu-layers-draft = 0`) works at any context but is slower. Field docs in `presets.rs`.
  - **GPU-device fields** (server-wide + per-preset model/draft) are dropdowns populated from `llama-server --list-devices` (`devices.rs`), probed once async at startup — not free text.
  - **CLI** (`cli.rs`, clap): `server` (show/set), `preset` (list/show/delete). No per-field preset `set`.
  - Reads/writes `%LOCALAPPDATA%\llama.cpp\config\server.ini` + `presets.ini` (`paths.rs`, `server_cfg.rs`, `presets.rs`, `ini.rs`).
  - Build-time ICO→PNG (`build.rs` + `ico` crate); `winresource` embeds the ICO as an EXE resource on Windows.
  - **Single-instance** (`single_instance.rs`, Windows-only Win32 FFI): a named mutex gates the GUI; a second launch pokes an activation event + `AllowSetForegroundWindow` and exits, and the live instance's listener thread surfaces its window. CLI subcommands unaffected.
- `resources\llama.ico` — icon used by the NSIS installer and embedded in `llama-cpp-config.exe`.

## Key conventions

- `common.ps1` is dot-sourced by `02-build.ps1` and `03-package.ps1`. Other scripts (`00-install-prerequisites.ps1`, `01-configure.ps1`) read `build\config-build.psd1` directly.
- `build\config-build.psd1` (build-time) uses single-quoted PowerShell-data-file syntax. Runtime config files (`server.ini`, `presets.ini`) are plain INI written as UTF-8 without BOM.
- All build artifacts (source clones, cmake binary dir, NSIS staging, sccache cache, `config-build.psd1`, generated `.nsi`, Rust build output copy) live under `.\build\` (gitignored). The final installer goes to `.\dist\`. `rm -rf build\` is a full clean wipe — it nukes the sccache too, so the next full build re-fills it from scratch.
- The installer registers in `HKLM\Software\llama.cpp` (InstallDir) and supports upgrades (uninstalls previous version first). User runtime state lives under `%LOCALAPPDATA%\llama.cpp\`: `config\server.ini` (single `[Server]` section), `config\presets.ini` (per-model presets consumed by `llama-server --models-preset`), `logs\llama-server.log`. The uninstaller asks (defaulting to No) whether to delete the whole tree, so it survives reinstall/upgrade by default.
- sccache is auto-detected; cache stored in `build\.sccache\`. Uses `SCCACHE_MAX_FRAME_LENGTH=104857600` for large GPU objects. nvcc is intentionally NOT wrapped with sccache — it mishandles multi-arch fatbin generation on CUDA 13.x.
- `llama-cpp-config` has zero Clippy warnings; uses `#[cfg]` compile-time branching for OS portability in `paths.rs`, `server_cfg.rs`, `gui.rs`, `model_scan.rs`, `runstate.rs`, and `proc.rs` (the shared `CREATE_NO_WINDOW` process launcher used by `devices.rs` / `server_version.rs` / `runstate.rs`).

## Common changes in `llama-cpp-config`

**Adding a per-model preset field** fans out to 7 places (kebab-case INI key ↔ snake_case Rust field). Tracing an existing field like `ctx-size` is the fastest template:

1. `Preset` struct field (+ doc comment) — `llama-cpp-config\src\presets.rs`
2. `impl Default for Preset` — same file
3. `Preset::from_keys` — INI read; `k.get("kebab-key")` → field
4. `render_section` — INI write; `emit_*(&mut out, "kebab-key", …)` + a `;` comment line
5. `PresetForm` struct — `llama-cpp-config\ui\types.slint`. Numeric-optional fields are carried as `string` in the form (e.g. `temp`, `spec_draft_n_max`); slider fields pair with an `_auto: bool`.
6. The input widget on the Models page — `llama-cpp-config\ui\models_page.slint`; bind it to `AppState.form.<field>` (state lives in `ui\state.slint`, not on the page). Editable widgets bind **two-way** (`<=>`), never one-way — a one-way binding goes stale after the first edit (guarded by `src\tests\ui_bindings.rs`, below).
7. `preset_to_form` + `form_to_preset` — `llama-cpp-config\src\form.rs`. Map `_auto` / blank ⇒ `None`; pull any real default from `Preset::default()` (via the local `d`) rather than re-hardcoding it, so the form and INI can't drift.

The `preset` CLI subcommand is list/show/delete only (no per-field `set`), so `cli.rs` needs no change. Round-trip is covered by the tests in `presets.rs` (INI render/parse) **and** `form.rs` (`form_to_preset(preset_to_form(p)) == p`) — the latter catches a field wired into one conversion but not the other. The binding *direction* of step 6 is covered by `llama-cpp-config\src\tests\ui_bindings.rs`: a headless Slint-testing-backend test that drives real widgets and fails if an editable one goes stale after an edit (the one-way-binding bug). Cross-cutting e2e tests live under `src\tests\` (an internal `#[cfg(test)]` module, not a top-level `tests\` dir — a binary crate exposes no library for an integration crate to link); per-module unit tests stay inline. It needs Slint element debug info, which `build.rs` emits for non-release builds only — so run it with `cargo test` (debug profile), not `cargo test --release`.
