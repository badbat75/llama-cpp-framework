# Project guide

## Project overview

Windows-only PowerShell scripts to install prerequisites, configure, build, and package [llama.cpp](https://github.com/ggerganov/llama.cpp) with CUDA, Vulkan, and ROCm (HIP) support, and NSIS installer packaging. Not compatible with Linux or macOS. llama.cpp's built-in chat interface (served by `llama-server`) is used as the frontend — no separate web UI is bundled.

A companion Rust binary (`llama-cpp-config`) provides a GUI + CLI configurator for `server.ini` and per-model `presets.ini`, replacing what were previously PowerShell config scripts.

## Structure

- `build\config-build.psd1` — build-time configuration (paths, GPU targets, compiler flags). Generated by `01-configure.ps1`; lives under `build\` (whole dir gitignored).
- `common.ps1` — shared bootstrap: loads `$cfg` from `build\config-build.psd1`, adds ROCm/HIP\bin to PATH, exposes `Enable-VsDevShell` as a function. VS Dev Shell activation is **opt-in**: only build/package scripts call it. Dot-sourced by `02-build.ps1` and `03-package.ps1`.
- `00-install-prerequisites.ps1` — idempotent toolchain bootstrapper: installs **and** updates the winget-tracked packages (PowerShell 7+ / OpenSSL / NSIS) in a single UAC-elevated session (`winget install` for missing, `winget upgrade` for present), then `git pull --ff-only`s `$cfg.LlamaCppDir` when cloned. Flags CUDA / Vulkan / AMD HIP for manual install. Reads `build\config-build.psd1` directly (does **not** dot-source `common.ps1`) so it works on a fresh machine before `01-configure.ps1` has run. End-of-run report shows before→after versions, plus a rebuild recommendation (`02-build.ps1` / `03-package.ps1`) only when llama.cpp source moved. Safe to re-run any time.
- `01-configure.ps1` — auto-detects paths, activates VS Dev Shell for tool discovery, verifies tools, creates `build\` if missing, writes `build\config-build.psd1`. Accepts `-LlamaCppDir` param (defaults to `$PSScriptRoot\build\llama.cpp`). Writes GPU targets (`gfx9xx`/`gfx1x`/`gfx12xx`), build type (Release), compiler (clang), march flags (`-march=x86-64-v3`), and build jobs.
- `02-build.ps1` — dot-sources `common.ps1` + calls `Enable-VsDevShell`. Builds **both** llama.cpp (CMake configure + Ninja) **and** `llama-cpp-config` (cargo build --release). Auto-clones/pulls llama.cpp into `build\llama.cpp\`, builds CMake output into `build\llama.cpp-cmake\`, and copies the Rust binary to `build\llama-cpp-config\`. Uses sccache if available for the C++ build. All under `.\build\` (gitignored).
- `03-package.ps1` — dot-sources `common.ps1` + calls `Enable-VsDevShell` (cmake --install needs the VS env). Stages from `build\llama.cpp-cmake\` + `build\llama-cpp-config\` into `build\staging\`, then builds NSIS installer to `.\dist\`. The package is named `llama-cpp-framework-v<frameworkVersion>-<llamaBuild>-<arch>-setup.exe`, e.g. `llama-cpp-framework-v1.0.0-b3456-x64-setup.exe`: **frameworkVersion** = the `llama-cpp-config` crate version (parsed from `llama-cpp-config\Cargo.toml` `[package]`; the configurator and the framework are versioned together), **llamaBuild** = `git describe --tags` of `$cfg.LlamaCppDir` (e.g. `b3456`), **arch** = `x64`/`arm64`.
- `llama-cpp.nsi.template` — NSIS installer template with placeholders (`@VERSION@` = framework/configurator version, `@LLAMA_BUILD@` = bundled llama.cpp build, `@STAGING_DIR@`, `@OUTPUT_FILE@`) replaced by `03-package.ps1` into `build\llama-cpp.nsi` (transient — deleted after the build). The ARP DisplayName/installer title read "llama.cpp-framework v@VERSION@ (llama @LLAMA_BUILD@)"; DisplayVersion = framework version; the llama build is also written to `HKLM\Software\llama.cpp\LlamaBuild`. Three components: binaries (required), PATH addition (optional), Start Menu shortcut (optional). Installs `bin\` (llama.cpp binaries, plus `llama-cpp-config.exe` and `llama.ico` side-by-side with `llama-server.exe`) and `lib\`. **No custom config page** — runtime configuration is handled entirely by `llama-cpp-config` post-install. Start Menu shortcut: "llama.cpp Config" → `llama-cpp-config.exe`.
- `llama-cpp-config\` — Rust GUI + CLI configurator (source). See [llama-cpp-config\README.md](llama-cpp-config\README.md).
  - GUI (Slint): Server tab (server.ini), Models tab (presets.ini), Integrations tab (opencode.json, Claude Code snippet), Model list with scan/filter, server status footer. The left nav rail carries the server run controls **at its bottom** (Start/Stop stacked over Open chat UI over Refresh — Start/Stop swaps on `server_running`, Open chat UI enabled only while running), so they are reachable from any tab. The **Refresh** button (always enabled) fires `reload_all` → `gui` re-reads `server.ini`/`presets.ini`, re-scans the models dir, and reloads integration state (`load_server_into_ui` + `refresh_presets` + `refresh_file_options` + `refresh_integrations` + `refresh_run_status`), picking up files added on disk or config files hand-edited outside the GUI without a restart. Stop is a three-state control: clicking it enters a transitional **"Stopping…"** (disabled) button held by the `server_stopping` flag while `gui::stop_server_async` runs `taskkill` off the UI thread and polls `runstate::load()` (tasklist) until the process actually disappears — a forced kill lingers a second or two as the GPU context unwinds — then flips to Start (wait capped at 15 s). Both the window and tray Stop paths funnel through `stop_server_async`. The Server tab's old "Control" card is now a read-only, multi-line **Command Line** card (Slint `TextEdit`): `runstate::command_line()` renders the exact `llama-server` invocation one `--flag value` per line, joined with the shell's line-continuation char (`` ` `` on Windows via `#[cfg(windows)]`, `\` on POSIX) so it pastes straight into a terminal. Between the **Assets** card (Model file / MMProj / Draft-model / Spec-type pickers) and the **Hardware Config** card, the Models tab shows a read-only **Model info** box (`model_info_*` props, refreshed via the `model_changed` callback → `gui::update_model_info`, also fired when the mmproj/draft pickers change). `gguf.rs` reads metadata with **llama.cpp's own gguf reader** — it runtime-loads `ggml-base.dll` (Win32 `LoadLibrary`/`GetProcAddress` FFI, like `single_instance.rs`; the DLL ships beside `llama-cpp-config.exe` in `bin\`) and calls `gguf_init_from_file` with `no_alloc` (header + tensor-infos only, no weights). No parser is reimplemented; a `KvSource` trait keeps field extraction testable via a map-backed source, and a `#[cfg(not(windows))]` `ffi::open` stub returns `None`. It reports dense vs MoE (+ `expert_count`/`expert_used_count`/`expert_shared_count`), and — for MoE models — a dedicated **MoE layers** row (`model_info_has_moe`/`model_info_moe`, from `ModelInfo::moe_offload_line`) that surfaces how many layers carry experts from the dense/MoE split (`leading_dense_block_count`/`moe_every_n_layers`), with a "saves VRAM (slower)" note — sizing the per-preset `n-cpu-moe` (`--n-cpu-moe`) field. It also reports `block_count`, trained `context_length`, GQA (`head_count`/`head_count_kv`), quant (`general.file_type` → name), and embedded MTP (`<arch>.nextn_predict_layers` > 0), plus a filename cross-reference against `mtps\`/`dflashs\`. When selected, a **MMProj** row reads the clip header (`clip.projector_type`, `clip.has_vision_encoder`/`has_audio_encoder`, `clip.vision.block_count`/`image_size`/`patch_size`) and a **Draft file** row reads the drafter's own header (arch, `block_count`, and DFlash `<arch>.block_size` → the `--spec-draft-n-max` ceiling of `block_size-1`). Reads are synchronous and uncached (parsing just the header is cheap enough for the UI thread); if the DLL can't be loaded there is no fallback — the box shows "unavailable". The Models tab groups the preset editor by concern rather than by feature. The **Assets** card holds the file pickers and speculator selection — Model file (`--model`), MMProj (`--mmproj`), ONE draft-model picker that merges MTP heads (scanned from `mtps\`) and DFlash drafters (scanned from `dflashs\`) — both feeding `--model-draft` — and the spec-type dropdown (`--spec-type`); the draft picker's parallel `draft_specs` array (built in `model_scan::build_draft_options`) auto-sets `--spec-type` to `draft-mtp` or `draft-dflash` on pick (and auto-pins the draft to the server GPU, else CPU). A dedicated **Hardware Config** card (directly under Model info) collects every placement knob: GPU device (`--device`), GPU split mode (`--split-mode`) + tensor split (`--tensor-split`), GPU layers (`--n-gpu-layers`), MoE CPU layers (`--n-cpu-moe`), draft device (`--device-draft`), and draft GPU layers (`--n-gpu-layers-draft`). The trimmed **Speculative decoding (MTP / DFlash)** card sits last (below Advanced) and carries only `--spec-draft-n-max` (preset key `spec-draft-n-max`; DFlash clamps it to the GGUF's trained `block_size-1`, e.g. 15). GPU-device fields (server-wide and the per-preset model/draft devices) are **dropdowns populated from `llama-server --list-devices`** (`devices.rs`), not free text — probed once async at startup (cached in `all_device_ids`/`all_device_labels`); each combobox keeps a `(default)` entry and preserves a stale/custom id as a `(custom)` row. Multi-GPU **weight distribution** is exposed both server-wide (`server.ini` `SplitMode`/`TensorSplit`, wired into the launch args by `runstate::server_args`) and per-preset (`presets.ini` `split-mode`/`tensor-split`, passed straight through `--models-preset`): `--split-mode` (none|layer|row) as a `default`-inheriting dropdown and `--tensor-split` (per-GPU proportions like `3,1`) as free text — identical on CUDA and HIP; a blank per-preset value inherits the server default. Several controls are **guardrailed** (Slint `enabled` bindings, no Rust involved): the speculative fields (spec-type, `--spec-draft-n-max`, draft device, draft GPU layers) are disabled unless a draft is available — a draft model is selected (`form.model_draft != ""`) **or** the main model embeds MTP heads (`model_info_embeds_mtp`, set from `nextn_predict_layers > 0`), combined into the `draft_active` binding; MoE CPU layers is disabled unless the model is MoE (`model_info_has_moe`, which also stays false when GGUF metadata is unavailable); and both `--split-mode` and `--tensor-split` — server-wide *and* per-preset — are disabled when a single GPU device is pinned (`server_device` / `form.device` non-empty), since a pinned device leaves nothing to split. The three layer-count fields — **GPU layers** (`--n-gpu-layers`), **MoE CPU layers** (`--n-cpu-moe`), **Draft GPU layers** (`--n-gpu-layers-draft`) — are Slint `Slider`s ranging `0..model_info_n_layer` / `0..model_info_draft_n_layer` (the model's / draft's `block_count`, published by `update_model_info`; 0 ⇒ unknown ⇒ a 0..99 fallback via `gpu_layers_max`/`draft_layers_max`). Each pairs with an **auto** checkbox (`*_auto` form bool) that omits the flag (maps to `None`) and disables the slider; all three default to auto for a new preset. Draft GPU layers additionally degrades to a plain CPU/GPU `Switch` (on = 99, off = 0) when `model_info_draft_n_layer == 0` — i.e. the draft is an MTP/nextn head with no transformer layers (`block_count 0`, shown as "0 layers (head)" in the Draft-file row), where a 0..N slider would be meaningless.
  - CLI: `server` (show/set), `preset` (list/show/delete) subcommands via clap.
  - Reads/writes `%LOCALAPPDATA%\llama.cpp\config\server.ini` and `presets.ini`.
  - Build-time ICO→PNG conversion (`build.rs` + `ico` crate); `winresource` embeds ICO as EXE resource on Windows.
  - Single-instance (`single_instance.rs`, Windows-only, raw Win32 FFI): a named mutex (`Local\llama-cpp-config.singleton`) gates the GUI. A second `llama-cpp-config.exe` launch finds the mutex taken, pokes a named auto-reset event (`…activate`) + `AllowSetForegroundWindow(ASFW_ANY)`, then exits; the live instance's listener thread surfaces its window (`show()` + winit `set_minimized(false)`/`focus_window()` via the `unstable-winit-030` accessor). CLI subcommands are unaffected.
- `resources\llama.ico` — icon used by the NSIS installer and embedded in `llama-cpp-config.exe`.

## Key conventions

- `common.ps1` is dot-sourced by `02-build.ps1` and `03-package.ps1`. Other scripts (`00-install-prerequisites.ps1`, `01-configure.ps1`) read `build\config-build.psd1` directly.
- `build\config-build.psd1` (build-time) uses single-quoted PowerShell-data-file syntax. Runtime config files (`server.ini`, `presets.ini`) are plain INI written as UTF-8 without BOM.
- All build artifacts (source clones, cmake binary dir, NSIS staging, sccache cache, `config-build.psd1`, generated `.nsi`, Rust build output copy) live under `.\build\` (gitignored). The final installer goes to `.\dist\`. `rm -rf build\` is a full clean wipe — it nukes the sccache too, so the next full build re-fills it from scratch.
- The installer registers in `HKLM\Software\llama.cpp` (InstallDir) and supports upgrades (uninstalls previous version first). User runtime state lives under `%LOCALAPPDATA%\llama.cpp\`: `config\server.ini` (single `[Server]` section), `config\presets.ini` (per-model presets consumed by `llama-server --models-preset`), `logs\llama-server.log`. The uninstaller asks (defaulting to No) whether to delete the whole tree, so it survives reinstall/upgrade by default.
- sccache is auto-detected; cache stored in `build\.sccache\`. Uses `SCCACHE_MAX_FRAME_LENGTH=104857600` for large GPU objects. nvcc is intentionally NOT wrapped with sccache — it mishandles multi-arch fatbin generation on CUDA 13.x.
- `llama-cpp-config` has zero Clippy warnings; uses `#[cfg]` compile-time branching for OS portability in `paths.rs`, `server_cfg.rs`, `gui.rs`, `model_scan.rs`, `runstate.rs`.
