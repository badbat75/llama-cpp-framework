# Project guide

## Project overview

Windows-only PowerShell scripts to install prerequisites, configure, build, and package [llama.cpp](https://github.com/ggerganov/llama.cpp) with CUDA, Vulkan, and ROCm (HIP) support, and NSIS installer packaging. Not compatible with Linux or macOS. llama.cpp's built-in chat interface (served by `llama-server`) is used as the frontend — no separate web UI is bundled.

A companion Rust binary (`llama-cpp-config`) provides a GUI + CLI configurator for `server.ini` and per-model `presets.ini`, replacing what were previously PowerShell config scripts.

## Structure

- `build\config-build.psd1` — build-time configuration (paths, GPU targets, compiler flags). Generated by `01-configure.ps1`; lives under `build\` (whole dir gitignored).
- `common.ps1` — shared bootstrap: loads `$cfg` from `build\config-build.psd1`, adds ROCm/HIP\bin to PATH, exposes `Enable-VsDevShell` as a function. VS Dev Shell activation is **opt-in**: only build/package scripts call it. Dot-sourced by `02-build.ps1` and `03-package.ps1`.
- `00-install-prerequisites.ps1` — idempotent toolchain bootstrapper: installs **and** updates the winget-tracked packages (PowerShell 7+ / OpenSSL / NSIS) in a single UAC-elevated session (`winget install` for missing, `winget upgrade` for present), then `git pull --ff-only`s `$cfg.LlamaCppDir` when cloned. Flags CUDA / Vulkan / AMD HIP for manual install. Reads `build\config-build.psd1` directly (does **not** dot-source `common.ps1`) so it works on a fresh machine before `01-configure.ps1` has run. End-of-run report shows before→after versions, plus a rebuild recommendation (`02-build.ps1` / `03-package.ps1`) only when llama.cpp source moved. Safe to re-run any time.
- `01-configure.ps1` — auto-detects paths, activates VS Dev Shell for tool discovery, verifies tools, creates `build\` if missing, writes `build\config-build.psd1`. Accepts `-LlamaCppDir` param (defaults to `$PSScriptRoot\build\llama.cpp`). Writes GPU targets (`gfx9xx`/`gfx1x`/`gfx12xx`), build type (Release), compiler (clang), march flags (`-march=x86-64-v3`), and build jobs.
- `02-build.ps1` — dot-sources `common.ps1` + calls `Enable-VsDevShell`. Builds **both** llama.cpp (CMake configure + Ninja) **and** `llama-cpp-config` (cargo build --release). Auto-clones/pulls llama.cpp into `build\llama.cpp\`, builds CMake output into `build\llama.cpp-cmake\`, and copies the Rust binary to `build\llama-cpp-config\`. Uses sccache if available for the C++ build. All under `.\build\` (gitignored).
- `03-package.ps1` — dot-sources `common.ps1` + calls `Enable-VsDevShell` (cmake --install needs the VS env). Stages from `build\llama.cpp-cmake\` + `build\llama-cpp-config\` into `build\staging\`, then builds NSIS installer to `.\dist\`.
- `llama-cpp.nsi.template` — NSIS installer template with placeholders (`@VERSION@`, `@STAGING_DIR@`, `@OUTPUT_FILE@`) replaced by `03-package.ps1` into `build\llama-cpp.nsi` (transient — deleted after the build). Three components: binaries (required), PATH addition (optional), Start Menu shortcut (optional). Installs `bin\` (llama.cpp binaries, plus `llama-cpp-config.exe` and `llama.ico` side-by-side with `llama-server.exe`) and `lib\`. **No custom config page** — runtime configuration is handled entirely by `llama-cpp-config` post-install. Start Menu shortcut: "llama.cpp Config" → `llama-cpp-config.exe`.
- `llama-cpp-config\` — Rust GUI + CLI configurator (source). See [llama-cpp-config\README.md](llama-cpp-config\README.md).
  - GUI (Slint): Server tab (server.ini), Models tab (presets.ini), Integrations tab (opencode.json, Claude Code snippet), Model list with scan/filter, server status footer. Models tab has a Speculative/MTP card (draft/MTP head picker scanned from `mtps\` + spec-type). GPU-device fields (server-wide and the per-preset model/draft devices) are **dropdowns populated from `llama-server --list-devices`** (`devices.rs`), not free text — probed once async at startup (cached in `all_device_ids`/`all_device_labels`); each combobox keeps a `(default)` entry and preserves a stale/custom id as a `(custom)` row.
  - CLI: `server` (show/set), `preset` (list/show/delete) subcommands via clap.
  - Reads/writes `%LOCALAPPDATA%\llama.cpp\config\server.ini` and `presets.ini`.
  - Build-time ICO→PNG conversion (`build.rs` + `ico` crate); `winresource` embeds ICO as EXE resource on Windows.
- `resources\llama.ico` — icon used by the NSIS installer and embedded in `llama-cpp-config.exe`.

## Key conventions

- `common.ps1` is dot-sourced by `02-build.ps1` and `03-package.ps1`. Other scripts (`00-install-prerequisites.ps1`, `01-configure.ps1`) read `build\config-build.psd1` directly.
- `build\config-build.psd1` (build-time) uses single-quoted PowerShell-data-file syntax. Runtime config files (`server.ini`, `presets.ini`) are plain INI written as UTF-8 without BOM.
- All build artifacts (source clones, cmake binary dir, NSIS staging, sccache cache, `config-build.psd1`, generated `.nsi`, Rust build output copy) live under `.\build\` (gitignored). The final installer goes to `.\dist\`. `rm -rf build\` is a full clean wipe — it nukes the sccache too, so the next full build re-fills it from scratch.
- The installer registers in `HKLM\Software\llama.cpp` (InstallDir) and supports upgrades (uninstalls previous version first). User runtime state lives under `%LOCALAPPDATA%\llama.cpp\`: `config\server.ini` (single `[Server]` section), `config\presets.ini` (per-model presets consumed by `llama-server --models-preset`), `logs\llama-server.log`. The uninstaller asks (defaulting to No) whether to delete the whole tree, so it survives reinstall/upgrade by default.
- sccache is auto-detected; cache stored in `build\.sccache\`. Uses `SCCACHE_MAX_FRAME_LENGTH=104857600` for large GPU objects. nvcc is intentionally NOT wrapped with sccache — it mishandles multi-arch fatbin generation on CUDA 13.x.
- `llama-cpp-config` has zero Clippy warnings; uses `#[cfg]` compile-time branching for OS portability in `paths.rs`, `server_cfg.rs`, `gui.rs`, `model_scan.rs`, `runstate.rs`.
