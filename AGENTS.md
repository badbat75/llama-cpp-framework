# Project guide

## Project overview

Windows-only PowerShell scripts to install prerequisites, configure, build, and package [llama.cpp](https://github.com/ggerganov/llama.cpp) with CUDA, Vulkan, and ROCm (HIP) support, and NSIS installer packaging. Not compatible with Linux or macOS. llama.cpp's built-in chat interface (served by `llama-server`) is used as the frontend — no separate web UI is bundled.

A companion Rust binary (`llama-cpp-config`) provides a GUI + CLI configurator for `server.ini` and per-model `presets.ini`, replacing what were previously PowerShell config scripts.

## Structure

- `build\config-build.psd1` — build-time configuration (paths, GPU targets, compiler flags). Generated by `01-configure.ps1`; lives under `build\` (whole dir gitignored).
- `common.ps1` — shared bootstrap: loads `$cfg` from `build\config-build.psd1`, adds ROCm/HIP\bin to PATH, exposes `Enable-VsDevShell` as a function. VS Dev Shell activation is **opt-in**: only build/package scripts call it. Dot-sourced by `02-build.ps1` and `03-package.ps1`.
- `00-install-prerequisites.ps1` — idempotent toolchain bootstrapper: installs **and** updates the winget-tracked packages (PowerShell 7+ / OpenSSL / NSIS) in a single UAC-elevated session (`winget install` for missing, `winget upgrade` for present, plus the OpenSSL lib-symlink fixup), then `git fetch`es `$cfg.LlamaCppDir` when cloned and compares the checked-out `bNNNN` tag against the newest one reachable from `origin/master` (no `git pull` — `02-build.ps1` keeps the clone on a detached tag, so a pull would always fail; the checkout onto the new tag is `02-build.ps1`'s job). Flags CUDA / Vulkan / AMD HIP for manual install. Reads `build\config-build.psd1` directly (does **not** dot-source `common.ps1`) so it works on a fresh machine before `01-configure.ps1` has run. End-of-run report shows before→after versions, plus a rebuild recommendation (`02-build.ps1` / `03-package.ps1`) only when a newer llama.cpp release tag is available. Safe to re-run any time.
- `01-configure.ps1` — auto-detects paths, activates VS Dev Shell for tool discovery, verifies tools, creates `build\` if missing, writes `build\config-build.psd1`. Accepts `-LlamaCppDir` param (defaults to `$PSScriptRoot\build\llama.cpp`). Writes GPU targets (`gfx9xx`/`gfx1x`/`gfx12xx`), build type (Release), compiler (clang), march flags (`-march=x86-64-v3`), and build jobs.
- `02-build.ps1` — dot-sources `common.ps1` + calls `Enable-VsDevShell`. Builds **both** llama.cpp (CMake configure + Ninja) **and** `llama-cpp-config` (cargo build --release). Auto-clones llama.cpp into `build\llama.cpp\` (or fetches if present) and **checks out its latest release tag** (`bNNNN`, the highest tag reachable from `origin/master`) rather than tracking master HEAD — so the build is a clean tagged release and `git describe` yields e.g. `b9867`, not `b9867-1-g…`. Builds CMake output into `build\llama.cpp-cmake\`; the Rust binary stays in cargo's `llama-cpp-config\target\release\` (no intermediate copy — `03-package.ps1` stages it from there). Uses sccache if available for the C++ build. For HIP it force-includes the checked-in `patches\__clang_hip_runtime_wrapper.h` (a ROCm/MSVC include-order workaround). All under `.\build\` (gitignored).
- `03-package.ps1` — dot-sources `common.ps1` + calls `Enable-VsDevShell` (cmake --install needs the VS env). Stages from `build\llama.cpp-cmake\` + `llama-cpp-config\target\release\` into `build\staging\`, then builds NSIS installer to `.\dist\`. The package is named `llama-cpp-framework-v<frameworkVersion>-<llamaBuild>-<arch>-setup.exe`, e.g. `llama-cpp-framework-v1.0.0-b3456-x64-setup.exe`: **frameworkVersion** = the `llama-cpp-config` crate version (parsed from `llama-cpp-config\Cargo.toml` `[package]`; the configurator and the framework are versioned together), **llamaBuild** = `git describe --tags` of `$cfg.LlamaCppDir` (e.g. `b3456`), **arch** = `x64`/`arm64`.
- `llama-cpp.nsi.template` — NSIS installer template with placeholders (`@VERSION@` = framework/configurator version, `@LLAMA_BUILD@` = bundled llama.cpp build, `@STAGING_DIR@`, `@OUTPUT_FILE@`) replaced by `03-package.ps1` into `build\llama-cpp.nsi` (transient — deleted after the build). The ARP DisplayName/installer title read "llama.cpp-framework v@VERSION@ (llama @LLAMA_BUILD@)"; DisplayVersion = framework version; the llama build is also written to `HKLM\Software\llama.cpp\LlamaBuild`. Three components: binaries (required), PATH addition (optional), Start Menu shortcut (optional). Installs `bin\` (llama.cpp binaries, plus `llama-cpp-config.exe` and `llama.ico` side-by-side with `llama-server.exe`) and `lib\`. **No custom config page** — runtime configuration is handled entirely by `llama-cpp-config` post-install. Start Menu shortcut: "llama.cpp Config" → `llama-cpp-config.exe`.
- `llama-cpp-config\` — Rust GUI + CLI configurator for `server.ini` + per-model `presets.ini`; three tabs (**Server** → `server.ini`, **Models** → `presets.ini`, **Integrations** → `opencode.json` model list + Claude Code snippet) plus llama-server run controls. This entry is only the **map**: the architecture, per-tab/per-card walkthrough, UI-binding conventions, source-layout table, and test topology all live in [llama-cpp-config\README.md](llama-cpp-config\README.md), and every module carries its own depth in a `//!` header comment — read those before diving in, not this file. Adding a preset field, a server field, or a UI property follows the checklists in **Common changes** below. One recurring footgun: gemma4 MTP heads (`n_layer=0`) crash under multi-device "auto" memory-fitting — pin model **and** draft to one device, or leave the draft on CPU (canonical write-up in the `presets.rs` field docs).
- `resources\llama.ico` — icon used by the NSIS installer and embedded in `llama-cpp-config.exe`.

## Key conventions

- `common.ps1` is dot-sourced by `02-build.ps1` and `03-package.ps1`. Other scripts (`00-install-prerequisites.ps1`, `01-configure.ps1`) read `build\config-build.psd1` directly.
- `build\config-build.psd1` (build-time) uses single-quoted PowerShell-data-file syntax. Runtime config files (`server.ini`, `presets.ini`) are plain INI written as UTF-8 without BOM.
- All build artifacts (source clones, cmake binary dir, NSIS staging, sccache cache, `config-build.psd1`, generated `.nsi`) live under `.\build\` (gitignored). The one exception is cargo's `llama-cpp-config\target\` (gitignored separately). The final installer goes to `.\dist\`. `rm -rf build\` wipes everything C++-side including the sccache — the next full build re-fills it from scratch — but leaves the Rust `target\` alone.
- The installer registers in `HKLM\Software\llama.cpp` (InstallDir) and supports upgrades (uninstalls previous version first). User runtime state lives under `%LOCALAPPDATA%\llama.cpp\`: `config\server.ini` (single `[Server]` section), `config\presets.ini` (per-model presets consumed by `llama-server --models-preset`), `logs\llama-server.log`. The uninstaller asks (defaulting to No) whether to delete the whole tree, so it survives reinstall/upgrade by default.
- sccache is auto-detected; cache stored in `build\.sccache\`. Uses `SCCACHE_MAX_FRAME_LENGTH=104857600` for large GPU objects. nvcc is intentionally NOT wrapped with sccache — it mishandles multi-arch fatbin generation on CUDA 13.x (`fatbinary` fails with "Could not open input file '<tu>.compute_75.ptx'" on every `.cu.obj`). Retested with sccache 0.16.0 (2026-07) — still broken, so host C/CXX (clang) caching only.
- Build parallelism (`config-build.psd1` `BuildJobs`, written by `01-configure.ps1`) defaults to **3/4 of logical cores** — for both the CMake/Ninja `-j` and the `cargo build --jobs` legs — leaving CPU headroom during a long CUDA/HIP compile. `BuildJobs = 0` means "use all cores".
- `llama-cpp-config` has zero Clippy warnings; uses `#[cfg]` compile-time branching for OS portability in `paths.rs`, `gui.rs`, `runstate.rs`, `main.rs`, `gguf\ffi.rs`, and `proc.rs` (the shared `CREATE_NO_WINDOW` process launcher used by `devices.rs` / `server_version.rs` / `runstate.rs`); `single_instance.rs` is Windows-only wholesale.

## Common changes in `llama-cpp-config`

The two recurring recipes live as authoritative checklists at the top of the module they start in — follow those, not a from-memory list:

- **Adding a per-model preset field** fans out to 7 places (kebab-case INI key ↔ snake_case Rust field; INI schema → form struct → widget → conversions). Checklist: top of `llama-cpp-config\src\presets.rs`. Trace an existing field like `ctx-size` as the template.
- **Adding a server (server.ini) field** is the same shape plus **three spots in `cli.rs`** (the server has a per-field `set`; the preset CLI is list/show/delete only) plus **`runstate::server_args`** (the launch/Command-Line leg). Checklist: top of `llama-cpp-config\src\server_cfg.rs`.

Cross-cutting rules the recipes rely on:

- Editable widgets bind **two-way** (`<=>`) to `AppState.…` — a one-way binding goes stale after the first edit. The sanctioned one-way exceptions (read-only displays, `for`-row widgets, `MappedComboBox`, the Slider/AutoSlider push, `SegmentedControl`) are catalogued in the crate README's conventions section.
- Nearly every step is guarded by a test that fails when skipped: INI and form round-trips (`presets.rs` / `form.rs`; `server_cfg.rs` / `server_form.rs`), the CLI leg's exhaustive-struct tests (`cli.rs`), the launch-arg leg (`runstate.rs`), widget-binding staleness (`src\tests\ui_bindings.rs` — one widget PER KIND, plus the one-way-binding text lint `src\tests\binding_lint.rs`) and the Models/Integrations callback funnel (`src\tests\save_flow.rs`). The unguarded remainder — nothing fails for either: forgetting the widget entirely (the field just never appears in the UI), and skipping the path-field `validate_for_save` step (a `;`/`#` path then saves fine and reloads truncated; see the checklists' final step).
- Run the e2e tests with `cargo test` (debug profile), **not** `--release` — they need Slint element debug info that `build.rs` emits for non-release builds only. Full test-topology rationale (single shared `#[test]`, `LLAMA_CPP_CONFIG_DATA_ROOT` redirect, `src\tests\` vs `tests\`): crate README "Tests" section.
