# Project guide

## Project overview

Windows-only PowerShell scripts to install prerequisites, configure, build, and package [llama.cpp](https://github.com/ggerganov/llama.cpp) with CUDA, Vulkan, and ROCm (HIP) support, and NSIS installer packaging. Not compatible with Linux or macOS. llama.cpp's built-in chat interface (served by `llama-server`) is used as the frontend — no separate web UI is bundled.

A companion Rust binary (`llama-cpp-config`) provides a GUI + CLI configurator for `server.ini` and per-model `presets.ini`, replacing what were previously PowerShell config scripts.

## Structure

- `build\config-build.psd1` — build-time configuration (paths, GPU targets, compiler flags). Generated by `01-configure.ps1`; lives under `build\` (whole dir gitignored).
- `common.ps1` — shared bootstrap: loads `$cfg` from `build\config-build.psd1`, adds ROCm/HIP\bin to PATH, exposes `Enable-VsDevShell` as a function. VS Dev Shell activation is **opt-in**: only build/package scripts call it. Dot-sourced by `02-build.ps1` and `03-package.ps1`.
- `00-install-prerequisites.ps1` — idempotent toolchain bootstrapper: installs **and** updates the winget-tracked packages (PowerShell 7+ / OpenSSL / NSIS) in a single UAC-elevated session (`winget install` for missing, `winget upgrade` for present, plus the OpenSSL lib-symlink fixup), then `git fetch`es `$cfg.LlamaCppDir` when cloned and compares the checked-out `bNNNN` tag against the newest one reachable from `origin/master` (no `git pull` — `02-build.ps1` keeps the clone on a detached tag, so a pull would always fail; the checkout onto the new tag is `02-build.ps1`'s job). Flags CUDA / Vulkan / AMD HIP for manual install. Reads `build\config-build.psd1` directly (does **not** dot-source `common.ps1`) so it works on a fresh machine before `01-configure.ps1` has run. End-of-run report shows before→after versions, plus a rebuild recommendation (`02-build.ps1` / `03-package.ps1`) only when a newer llama.cpp release tag is available. Safe to re-run any time.
- `01-configure.ps1` — auto-detects paths, activates VS Dev Shell for tool discovery, verifies tools, creates `build\` if missing, writes `build\config-build.psd1`. Accepts `-LlamaCppDir` param (defaults to `$PSScriptRoot\build\llama.cpp`). Writes GPU targets (`gfx9xx`/`gfx1x`/`gfx12xx`), build type (Release), compiler (clang), march flags (`-march=x86-64-v3`), and build jobs.
- `02-build.ps1` — dot-sources `common.ps1` + calls `Enable-VsDevShell`. Builds **both** llama.cpp (CMake configure + Ninja) **and** `llama-cpp-config` (cargo build --release). Auto-clones llama.cpp into `build\llama.cpp\` (or fetches if present) and **checks out its latest release tag** (`bNNNN`, the highest tag reachable from `origin/master`) rather than tracking master HEAD — so the build is a clean tagged release and `git describe` yields e.g. `b9867`, not `b9867-1-g…`. Builds CMake output into `build\llama.cpp-cmake\`; the Rust binary stays in cargo's `llama-cpp-config\target\release\` (no intermediate copy — `03-package.ps1` stages it from there). Uses sccache if available for the C++ build. For HIP it force-includes the checked-in `patches\__clang_hip_runtime_wrapper.h` (a ROCm/MSVC include-order workaround). All under `.\build\` (gitignored).
- `03-package.ps1` — dot-sources `common.ps1` + calls `Enable-VsDevShell` (cmake --install needs the VS env). Stages from `build\llama.cpp-cmake\` + `llama-cpp-config\target\release\` into `build\staging\`, then builds NSIS installer to `.\dist\`. The package is named `llama-cpp-framework-v<frameworkVersion>-<llamaBuild>-<arch>-setup.exe`, e.g. `llama-cpp-framework-v1.0.0-b3456-x64-setup.exe`: **frameworkVersion** = the `llama-cpp-config` crate version (parsed from `llama-cpp-config\Cargo.toml` `[package]`; the configurator and the framework are versioned together), **llamaBuild** = `git describe --tags` of `$cfg.LlamaCppDir` (e.g. `b3456`), **arch** = `x64`/`arm64`.
- `llama-cpp.nsi.template` — NSIS installer template with placeholders (`@VERSION@` = framework/configurator version, `@LLAMA_BUILD@` = bundled llama.cpp build, `@STAGING_DIR@`, `@OUTPUT_FILE@`) replaced by `03-package.ps1` into `build\llama-cpp.nsi` (transient — deleted after the build). The ARP DisplayName/installer title read "llama.cpp-framework v@VERSION@ (llama @LLAMA_BUILD@)"; DisplayVersion = framework version; the llama build is also written to `HKLM\Software\llama.cpp\LlamaBuild`. Three components: binaries (required), PATH addition (optional), Start Menu shortcut (optional). Installs `bin\` (llama.cpp binaries, plus `llama-cpp-config.exe` and `llama.ico` side-by-side with `llama-server.exe`) and `lib\`. **No custom config page** — runtime configuration is handled entirely by `llama-cpp-config` post-install. Start Menu shortcut: "llama.cpp Config" → `llama-cpp-config.exe`.
- `llama-cpp-config\` — Rust GUI + CLI configurator for `server.ini` + per-model `presets.ini`; three tabs (**Server** → `server.ini`, **Models** → `presets.ini`, **Integrations** → `opencode.json` model list + Claude Code snippet) plus llama-server run controls. This entry is only the **map**: the architecture, per-tab/per-card walkthrough, UI-binding conventions, source-layout table, and test topology all live in [llama-cpp-config\README.md](llama-cpp-config\README.md), and every module carries its own depth in a `//!` header comment — read those before diving in, not this file. Adding a preset field, a server field, or a UI property follows the checklists in **Common changes** below. Nine recurring footguns, each with a canonical write-up in the code:
  - **GPU placement is a table, not two text fields** (`gpu_split.rs` + `GpuSplitTable`): `--tensor-split` is positional over the devices in `--device`, in that order, so neither means anything alone. A blank tensor-split over 2+ GPUs = split by *free VRAM*, not evenly.
  - **"All layers offloaded" still leaves the embedding table in PINNED HOST RAM** (`tensor_override.rs` + `TensorOverrideTable`): llama.cpp parks `token_embd.weight` in a host buffer even at `offloaded N/N layers to GPU` — and with a GPU backend active that buffer is pinned (`ROCm_Host`/`CUDA_Host` in the log), which Windows reports as **Shared GPU memory**. So a model that fits in VRAM with room to spare still shows GBs of shared allocation. The cure is an `override-tensor` rule (`token_embd\.weight=ROCm0`); the value's grammar is split on `,` and at the first `=` *before* parsing and cannot be escaped, which is why it is a table and not a text field.
  - **…but that cure is CONDITIONAL, and violating either condition costs 10x** (`gguf.rs` `embd_line` / `embd_pin_warning`, surfaced as Model info → **Embeddings** + a warning strip on the table): the pin is **not model-agnostic**, which is the natural (and wrong) reading of the bullet above. (1) `token_embd`'s **own** quant must have a GPU `get_rows` kernel — ggml-cuda/ggml-hip whitelist `F32/F16/BF16/I32/Q1_0/Q4_0/Q4_1/Q5_0/Q5_1/Q8_0` and `default: return false` for **every K-quant and IQ-quant** (`ggml-cuda.cu`, `supports_op`). Pin a K-quant table and it sits in VRAM the GPU cannot read, so the lookup round-trips to the host *every token*: measured 25 → **2.8 tok/s** (Q4_K embd), → **1.9 tok/s** (Q5_K — bigger table, worse). Prefill is untouched (~400 tok/s — one round-trip per 2048-token batch amortizes it) and the GPU engines read **0%**, so it presents as an *idle* GPU, not a slow one. The file NAME does not tell you: Unsloth's `Q6_K_XL` keeps `token_embd` at **Q8_0** (safe) while its `Q4_K_XL`/`Q5_K_XL` drop it to Q4_K/Q5_K — same model, same build, opposite outcome. (2) The card must have **room**: pinning MOVES the table from host RAM into VRAM (~1.26 GiB at Q8_0), a straight add that tips an already-full card into paging (Q6_K + f16 KV @ 256k ctx: 23 → **13.7 tok/s** *with* the pin). Because condition (1) is per-model, prefer the **per-preset** scope over the server-wide one. `gpu_has_get_rows` mirrors that upstream switch and must be re-checked when bumping llama.cpp — a type that gains a kernel and is missing here only costs a needless warning, but a type listed here *without* one green-lights the pin this exists to prevent.
  - **A server-wide `--override-tensor` REPLACES each preset's, it does not add to it** (`server_cfg.rs` `override_tensor`): the same table exists in both scopes, and `-ot` `push_back`s onto a vector inside llama.cpp — so "the two lists concatenate" is the natural guess and it is wrong. The router merges its own CLI args into every preset as a KEY→VALUE map (`common_preset::merge` → `options[opt] = val`, CLI as the writer), so exactly one `--override-tensor` reaches a child: the server's. Verified against b9976. Two consequences: the Models tab must say it is shadowed (`tensor_override_warning`), and the buffer type is validated by the ROUTER at startup — an unknown device in the server-wide value fails with `unknown buffer type` and **nothing starts**, where the same typo in a preset only kills that model.
  - **The image encoder ignores `--device`** (`server_cfg.rs` `mmproj_device`): `clip_ctx` takes the first GPU backend the registry offers unless `MTMD_BACKEND_DEVICE` names one, so on a mixed box it holds VRAM on the wrong card and computes only on image requests — which reads exactly like an idle GPU that has been "assigned something".
  - **Preserving past thinking is ONE flag, not a template kwarg** (`presets.rs` `reasoning_preserve` + the crate README): `--reasoning-preserve` sets the kwarg `preserve_reasoning`, which `caps_apply_preserve_reasoning` (`common/jinja/caps.cpp`) expands into THREE template variables at once — `preserve_thinking`, `clear_thinking`, `truncate_history_thinking` — because templates disagree on which they read (Qwen3.6/LFM2.5 the first, GLM-4.7 the second, Nemotron the third). Hand-writing one of them into `chat-template-kwargs` sets one of the three: it works on the templates using that name and is a **silent no-op** on the others, and it never registers with the capability probe, so llama-server still prints `consider enabling it via --reasoning-preserve`. Note "supported" only means the trace showed up in the probe's render — a template that preserves *unconditionally* (Ornith-1.0) reads as supported while gating on nothing.
  - **`--parallel` DIVIDES the context, and omitting it is not "1 slot"** (`integrations.rs` `effective_ctx`, surfaced as each opencode model's `limit.context`): the context one request may use is `n_ctx_seq = n_ctx / n_parallel` (`llama-context.cpp`) — so `parallel = 2` halves it — **unless the key is absent**, because llama-server defaults `n_parallel` to `-1` = auto, and auto picks 4 slots *and a unified KV cache* (`tools/server/server.cpp`), under which every slot sees the whole `n_ctx`. Compounding it: `ctx-size` absent means the model's **trained** context (`n_ctx = 0` → `hparams.n_ctx_train`), not a fixed default. Both together are why opencode's advertised context is neither `ctx-size` nor the trained context, and why the naive fix is *worse* than the bug it replaces — publishing 256k for a 256k model that pins `parallel = 2` promises a context llama-server rejects at 128k+.
  - **Thinking support is read from the template's SOURCE, and the verdict is about the TEMPLATE** (`gguf.rs` `Thinking` / `thinking_line` / `past_thinking_line`, surfaced as Model info → **Thinking** + **Past thinking**): there is no Jinja engine here, while llama.cpp answers the same question by *rendering* the template (`caps_get`, `common_chat_templates_support_enable_thinking`). So the scan looks for the names the flags act on — `enable_thinking` for `--reasoning`, and `preserve_thinking`/`clear_thinking`/`truncate_history_thinking` for `--reasoning-preserve` — which makes a `yes` a fact and a `no` only "no marker this build knows about" (MiMo-VL's template names nothing yet the model thinks; llama-server parses `<think>` out of the *output* regardless). The **Past thinking** row is nonetheless the one thing llama-server's log CANNOT tell you: its probe prints `supports preserving reasoning` for a template that gates on the flag *and* for one that preserves unconditionally (Ornith-1.0), which are opposite answers to "will this flag do anything".
  - **Draft placement needs a draft FILE** (`presets.rs` field docs): llama.cpp reads `--device-draft` / `--n-gpu-layers-draft` only inside `if (has_dft())`. With MTP heads embedded in the main GGUF the draft context is built against the target model and runs on its device, ignoring both. With a SEPARATE MTP head file (gemma4-assistant, `n_layer=0`) they do apply — and there, pin model **and** draft to one device, or leave the draft on CPU: multi-device "auto" memory-fitting crashes those heads.
- `resources\llama.ico` — icon used by the NSIS installer and embedded in `llama-cpp-config.exe`. **Generated, not checked in** (gitignored): `resources\generate-llama-ico.mjs` rasterizes it from llama.cpp's webui logo (`tools\ui\src\lib\assets\logo.svg`, a `currentColor` SVG) as a `#111111` glyph on a white rounded tile, matching how upstream renders its own PWA/apple-touch icons (upstream commits no `.ico` — its `favicon.ico` comes from the webui npm build). `llama-cpp-config\build.rs` (and, belt-and-braces, `03-package.ps1`) runs the script automatically whenever the file is missing; it needs node+npm and the llama.cpp clone, so a fresh checkout needs `02-build.ps1` (which clones) before a bare `cargo build`/`cargo test` can succeed. Delete `llama.ico` to force a regeneration after an upstream logo change.
- `tools\report-memory.ps1` — diagnostic (not part of the build pipeline; reads only `%LOCALAPPDATA%\llama.cpp\`, so it runs on an installed machine with no repo or `config-build.psd1`). Slices the memory of the **currently running** llama-server across two sources and cross-checks them: llama-server's own load log (every buffer announces its size and device → the *requested* footprint, rebuilt exactly) and Windows' `GPU Process Memory` perf counters (what the driver *actually* handed out: dedicated VRAM vs shared system RAM). The divergence is the point — an oversubscribed device does not fail to allocate, WDDM silently pages the excess over PCIe and the only symptom is slow decoding. Each balance sheet folds tensor placement in as an **`of which tensor override`** sub-row under `model weights` — never a row of its own, and never summed: an `--override-tensor` rule *moves* a tensor into that device's model buffer, it allocates nothing extra, so it is already inside `model weights` and inside `requested`, and totalling it again would inflate the one figure the whole report exists to compare against `free at load`. The host section additionally names the pinned share of the model weights, which is the shared-GPU-memory number and the reason the tensor-placement table exists at all. `-Json` for machine output.

  Two sources for the placement leg, and neither is the obvious one. The **rules** come from the ROUTER's `spawning server instance with args:` block, not from `presets.ini`/`server.ini`: the router overlays its own CLI args onto every preset, so the files do not say what actually ran, and the spawn line is the already-merged winner. That block sits *outside* the load block (above the `device_info` banner), and it is matched to the child by the `[port]` prefix every child line carries, never by "the spawn header nearest above" — with several models up, that would pick the wrong one. The **hits** come from `tensor <name> (<n> MiB <type>) buffer type overridden to <buft>`, which is `LLAMA_LOG_DEBUG` — i.e. verbosity **5** (`common/log.h`: `LOG_LEVEL_DEBUG = 5`, printed when `verbosity <= thold`), so at the usual `-lv 4` it is absent and *no hits is not the same as nothing matched*; the report says which.

  Seven parsing traps, all handled. llama.cpp **re-emits** `sched_reserve` for the draft context *after* the mmproj loads (so clip ownership must come from the `reserve_compute_meta` source line, never from a "seen the clip banner" latch) and **re-reserves** the compute graph once the slots come up (same buffer announced twice → collapse per `(owner, kind, device)`, do not sum). The log accounts for no driver context (a few hundred MiB per backend), so `requested` sitting just under `free at load` can still spill — trust the live counter there. With `--fit on` the model is **loaded twice** (a `no_alloc` dry run to size the fit, then the real one) and every override line is emitted in both passes → collapse by tensor name or every rule double-counts. `CPU_Mapped` — the buffer holding mmap'd weights, i.e. the whole CPU-side model unless `--no-mmap` — is **host memory, not a device**: matching host on `^CPU$` (rather than `^CPU`) invents a phantom GPU called CPU_Mapped and charges the model to it. A **sliding-window model builds two KV caches** (`llama_kv_cache_iswa`: a full one for the global-attention layers, a small one for the SWA layers) and each announces itself with the *same* `llama_kv_cache: <dev> KV buffer size` line — same source, same device, same kind — so the de-duplication above silently eats one of them (Gemma-4-12B: 816 MiB kept, 255 MiB dropped, and a card that was 4 MiB from full read as comfortable); the cure is a latch on the `creating non-SWA / creating SWA KV cache` banner, which is safe here precisely where it was not for the clip graph, because llama.cpp prints it immediately before building each cache, in both passes. A **separate draft/MTP head prints its own `print_info` block after the model's**, so arch/params/file type/file size must take the FIRST occurrence — otherwise a gemma4 + gemma4-assistant pair reports the head's arch and its nonsense parameter count (`422.86 B params` for a 12B model). And the set of `buffer size` lines llama.cpp can print is exactly **model / KV / RS / compute / output / LoRA / DSV4 `<name>` state** — the regex must be exhaustive, because a kind it does not match does not come out small, it vanishes from the balance sheet altogether (LoRA adapters and DeepSeek-V4 state were doing exactly that).

  One asymmetry the report has to encode rather than infer: a rule ending in **`=CPU` does not mean ordinary RAM**. With a GPU backend up and mmap off, llama.cpp serves CPU-side weights out of that backend's *pinned* host buffer (`CUDA_Host`/`ROCm_Host`) — which Windows charts under the card as **shared GPU memory**, so the VRAM is freed and the shared allocation is not. With mmap on, the same debug line is logged and the tensor is then served straight from the mapped file (`CPU_Mapped`, ordinary RAM), leaving the named pinned buffer at 0 MiB. Same log line, opposite outcome. Hence two rules in the code: the pinned verdict is gated on a pinned model buffer actually appearing in the `buffer size` lines, never on the override line alone; and an `of which` that comes out **larger than the row it hangs off** is not rounding, it is that contradiction — the sub-row is suppressed and the mismatch reported instead of printing `1,302 MiB of which` against a `0.00 MiB` row.

## Key conventions

- `common.ps1` is dot-sourced by `02-build.ps1` and `03-package.ps1`. Other scripts (`00-install-prerequisites.ps1`, `01-configure.ps1`) read `build\config-build.psd1` directly.
- `build\config-build.psd1` (build-time) uses single-quoted PowerShell-data-file syntax. Runtime config files (`server.ini`, `presets.ini`) are plain INI written as UTF-8 without BOM.
- All build artifacts (source clones, cmake binary dir, NSIS staging, sccache cache, `config-build.psd1`, generated `.nsi`) live under `.\build\` (gitignored). The one exception is cargo's `llama-cpp-config\target\` (gitignored separately). The final installer goes to `.\dist\`. `rm -rf build\` wipes everything C++-side including the sccache — the next full build re-fills it from scratch — but leaves the Rust `target\` alone.
- The installer registers in `HKLM\Software\llama.cpp` (InstallDir) and supports upgrades (uninstalls previous version first). User runtime state lives under `%LOCALAPPDATA%\llama.cpp\`: `config\server.ini` (single `[Server]` section), `config\presets.ini` (per-model presets consumed by `llama-server --models-preset`), `logs\llama-server.log`. The uninstaller asks (defaulting to No) whether to delete the whole tree, so it survives reinstall/upgrade by default.
- sccache is auto-detected; cache stored in `build\.sccache\`. Uses `SCCACHE_MAX_FRAME_LENGTH=104857600` for large GPU objects. nvcc is intentionally NOT wrapped with sccache — it mishandles multi-arch fatbin generation on CUDA 13.x (`fatbinary` fails with "Could not open input file '<tu>.compute_75.ptx'" on every `.cu.obj`). Retested with sccache 0.16.0 (2026-07) — still broken, so host C/CXX (clang) caching only.
- HIP GPU targets (`config-build.psd1` `GpuTargets` → `-DGPU_TARGETS`) are two groups, and each arch costs ~56 MiB of `ggml-hip.dll` plus compile time. **Inference-capable**: `gfx906;gfx1030;gfx1100;gfx1101;gfx1102;gfx1150;gfx1151;gfx1200;gfx1201` — exactly the archs the Windows ROCm 7.1 rocBLAS ships Tensile kernels for (`bin\rocblas\library\TensileLibrary_lazy_gfx*.dat` is the authoritative list; CDNA/Vega `gfx900/908/90a/942/950` is deliberately absent — no Windows kernels, no Windows driver for MI cards). **Code-object-only**: `gfx1035;gfx1036;gfx1103;gfx1152;gfx1153` (Ryzen iGPUs) — useless for inference but mandatory in the fatbin, because the driver-bundled HIP runtime (`amdhip64_7.dll` 10.0.3679) fails kernel-module load on *every* visible device when one of them has no matching code object: a visible iGPU otherwise kills the dGPU with "device kernel image is invalid". Do not prune either group. Related footgun: ROCm's `hip-config-amd.cmake` derives the `--offload-arch` flags from **`GPU_BUILD_TARGETS`**, which it seeds from `GPU_TARGETS` via `set(... CACHE ...)` — a no-op once cached, so editing `GpuTargets` silently does nothing on an existing `build\llama.cpp-cmake\`. `02-build.ps1` passes `-UGPU_BUILD_TARGETS` to clear it every configure.
- Build parallelism (`config-build.psd1` `BuildJobs`, written by `01-configure.ps1`) defaults to **3/4 of logical cores** — for both the CMake/Ninja `-j` and the `cargo build --jobs` legs — leaving CPU headroom during a long CUDA/HIP compile. `BuildJobs = 0` means "use all cores".
- `llama-cpp-config` has zero Clippy warnings; uses `#[cfg]` compile-time branching for OS portability in `paths.rs`, `gui.rs`, `runstate.rs`, `main.rs`, `gguf\ffi.rs`, and `proc.rs` (the shared `CREATE_NO_WINDOW` process launcher used by `devices.rs` / `server_version.rs` / `runstate.rs`); `single_instance.rs` is Windows-only wholesale.

## Common changes in `llama-cpp-config`

The two recurring recipes live as authoritative checklists at the top of the module they start in — follow those, not a from-memory list:

- **Adding a per-model preset field** fans out to 7 places (kebab-case INI key ↔ snake_case Rust field; INI schema → form struct → widget → conversions). Checklist: top of `llama-cpp-config\src\presets.rs`. Trace an existing field like `ctx-size` as the template.
- **Adding a server (server.ini) field** is the same shape plus **three spots in `cli.rs`** (the server has a per-field `set`; the preset CLI is list/show/delete only) plus **`runstate::server_args`** (the launch/Command-Line leg). Checklist: top of `llama-cpp-config\src\server_cfg.rs`.

Cross-cutting rules the recipes rely on:

- Editable widgets bind **two-way** (`<=>`) to `AppState.…` — a one-way binding goes stale after the first edit. The sanctioned one-way exceptions (read-only displays, `for`-row widgets, `MappedComboBox`, the Slider/AutoSlider push, `SegmentedControl`) are catalogued in the crate README's conventions section.
- Nearly every step is guarded by a test that fails when skipped: INI and form round-trips (`presets.rs` / `form.rs`; `server_cfg.rs` / `server_form.rs`), the CLI leg's exhaustive-struct tests (`cli.rs`), the launch-arg leg (`runstate.rs`), widget-binding staleness (`src\tests\ui_bindings.rs` — one widget PER KIND, plus the one-way-binding text lint `src\tests\binding_lint.rs`) and the Models/Integrations callback funnel (`src\tests\save_flow.rs`). The unguarded remainder — nothing fails for either: forgetting the widget entirely (the field just never appears in the UI), and skipping the path-field `validate_for_save` step (a `;`/`#` path then saves fine and reloads truncated; see the checklists' final step).
- Run the e2e tests with `cargo test` (debug profile), **not** `--release` — they need Slint element debug info that `build.rs` emits for non-release builds only. Full test-topology rationale (single shared `#[test]`, `LLAMA_CPP_CONFIG_DATA_ROOT` redirect, `src\tests\` vs `tests\`): crate README "Tests" section.

## Release process

1. **Bump version** in `llama-cpp-config\Cargo.toml` (`[package] version = "X.Y.Z"`). The next build also refreshes the crate's line in `Cargo.lock` — both files belong to the release commit.
2. **Run `02-build.ps1`** — builds llama.cpp (fetches latest `bNNNN` tag) and llama-cpp-config.
3. **Run `03-package.ps1`** — stages binaries, builds NSIS installer to `.\dist\llama-cpp-framework-vX.Y.Z-bNNNN-x64-setup.exe`.
4. **Commit** all changes. Subject: **`Release vX.Y.Z: <summary>`** — the SAME first line as the tag (step 5); every release commit follows this (`git log --oneline` — a `feat:`/`docs:`-style subject is not the convention here). Body: a prose summary, closing with a **`Bundles llama.cpp bNNNN (up from bMMMM).`** line.
5. **Tag**: `git tag -a X.Y.Z -m "Release vX.Y.Z: <short summary>..." && git push origin X.Y.Z`
   - Tag name: **no `v` prefix** (e.g. `1.5.1`, not `v1.5.1`).
   - Tag message: starts with `Release vX.Y.Z: <summary>` on first line, then blank line, then bullet points of changes. Match the style of previous tags (`git show 1.5.0`).
6. **Create release**: `gh release create X.Y.Z --title "vX.Y.Z (llama.cpp bNNNN)" --notes "<markdown body>"`
   - Title format: **`vX.Y.Z (llama.cpp bNNNN)`** (with `v` prefix, includes bundled llama.cpp build).
   - Body: detailed markdown with sections (`##`, `###`), bullet points, code formatting. Match the style of previous releases (`gh release view 1.5.0`). End with test count and bundled llama.cpp build.
7. **Attach installer**: `gh release upload X.Y.Z dist\llama-cpp-framework-vX.Y.Z-bNNNN-x64-setup.exe`

**Formatting conventions** (verified against 1.5.0):

| Artifact | Format | Example |
|---|---|---|
| Commit subject | `Release vX.Y.Z: <summary>` (matches the tag) | `Release v1.5.1: proxy/gateway support...` |
| Tag name | `X.Y.Z` (no `v`) | `1.5.1` |
