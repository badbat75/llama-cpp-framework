# Project guide

## Project overview

Windows-only PowerShell scripts to install prerequisites, configure, build, and run [llama.cpp](https://github.com/ggerganov/llama.cpp) with CUDA, Vulkan, and ROCm (HIP) support, optional Open WebUI frontend, and NSIS installer packaging. Not compatible with Linux or macOS.

## Structure

- `config-build.psd1` — build-time configuration (paths, GPU targets, compiler flags). Generated by `01-configure.ps1`, gitignored. Runtime / per-model settings are **not** stored at repo root — they live under `%LOCALAPPDATA%\llama.cpp\config\` and are written by `resources\config-server.ps1` / `config-model.ps1` on first launch (or by the NSIS install-time pages).
- `common.ps1` — shared bootstrap: loads `$cfg` from `config-build.psd1`, adds ROCm/HIP\bin to PATH, exposes `Enable-VsDevShell` as a function. VS Dev Shell activation is **opt-in**: only build/package scripts call it. Runtime scripts (`04-run.ps1`) dot-source `common.ps1` only for the ROCm PATH setup.
- `00-install-prerequisites.ps1` — installs PowerShell 7+ (`pwsh`, required by the runtime shortcuts) and OpenSSL via winget; flags CUDA / Vulkan SDK / AMD HIP for manual install. Self-elevates via UAC; combines all winget installs into a single elevated session.
- `01-configure.ps1` — auto-detects paths, verifies tools, writes `config-build.psd1`. Accepts `-LlamaCppDir` param (defaults to `.\build\llama.cpp`). Also defaults `OpenWebUIDir` to `.\build\open-webui`.
- `02-build.ps1` — dot-sources `common.ps1` + calls `Enable-VsDevShell`. CMake configure (out-of-source) + build with Ninja. Auto-clones/pulls llama.cpp into `build\llama.cpp\`, builds into `build\llama.cpp-cmake\`. Uses sccache if available. All under `.\build\` (gitignored).
- `02-build-webui.ps1` — dot-sources `common.ps1` for paths (no VS shell needed). Clones Open WebUI into `build\open-webui\`, builds Svelte frontend (Node.js 18-22 via fnm), installs from source into Python 3.11/3.12 venv at `build\webui-venv\`. Skips frontend build if already done for current commit. Recreates venv on repo update.
- `03-package.ps1` — dot-sources `common.ps1` + calls `Enable-VsDevShell` (cmake --install needs the VS env). Stages from `build\llama.cpp-cmake\` into `build\staging\`, then builds NSIS installer to `.\dist\`. Open WebUI installs to `C:\Program Files\Open WebUI` (separate from llama.cpp).
- `04-run.ps1` — dev-time launcher. Thin wrapper: dot-sources `common.ps1` for ROCm PATH, then delegates to `resources\run-model.ps1 -ServerExe .\build\llama.cpp-cmake\bin\llama-server.exe`. Pass `-WithWebUI` to also launch `resources\start-webui.ps1 -WebUIExe .\build\webui-venv\Scripts\open-webui.exe` in the background (killed via `taskkill /T` on exit). All arg-building logic lives in the `resources/` scripts so dev and installed runtimes can't drift.
- `05-update-framework.ps1` — maintenance script: `winget upgrade` for tracked tooling (PowerShell, OpenSSL, Python 3.12, fnm, NSIS), `git pull --ff-only` on llama.cpp and Open WebUI repos, and a final report of what changed. Recommends `02-build.ps1` / `02-build-webui.ps1` / `03-package.ps1` only when llama.cpp or Open WebUI sources moved (the only deps that get baked into the shipped binary). Manual SDKs (CUDA, Vulkan, ROCm) are not touched — only flagged.
- `resources/run-model.ps1` — runtime entry point shipped with the installer (main "llama.cpp" Start Menu shortcut). Reads `server.psd1` + `models\<ActiveModel>.psd1`; auto-invokes `config-server.ps1` / `config-model.ps1` if missing. Starts `llama-server` only. Accepts `-ServerExe` (defaults to `$PSScriptRoot\bin\llama-server.exe`) so `04-run.ps1` can point it at `.\build\llama.cpp-cmake\bin\` in dev mode. Tees stdout+stderr to `%LOCALAPPDATA%\llama.cpp\logs\llama-server.log`. CWD is forced to `%LOCALAPPDATA%\llama.cpp` so first-run files don't land in Program Files.
- `resources/start-webui.ps1` — standalone Open WebUI launcher (Start Menu shortcut "Start Open WebUI"). Reads `webui.psd1` for its own bind/port and `server.psd1` for the llama-server endpoint to probe. Accepts `-WebUIExe` so `04-run.ps1` can point it at `.\build\webui-venv\Scripts\open-webui.exe`. Sets `DATA_DIR` to `%LOCALAPPDATA%\llama.cpp\data` (writable per-user; required because Open WebUI's default falls back inside Program Files and crashes ChromaDB). Tees stdout+stderr to `%LOCALAPPDATA%\llama.cpp\logs\open-webui.log`. Doesn't auto-start llama-server.
- `resources/config-server.ps1` — llama-server config writer. Writes `%LOCALAPPDATA%\llama.cpp\config\server.psd1`. Holds only machine-wide knobs (Port, Hostname, Mlock, Threads, CacheReuse, ThreadsBatch); model-dependent params live in `models\<id>.psd1`. NSIS calls it with `-NonInteractive -Port -Hostname` from the install-time page. Preserves `ModelsDir` / `ActiveModel` set by config-model.ps1. Re-runnable via "Configure llama-server" Start Menu shortcut.
- `resources/config-webui.ps1` — Open WebUI config writer. Writes `webui.psd1` (Port + Hostname). Same interactive / `-NonInteractive` dual mode as config-server.ps1.
- `resources/config-model.ps1` — per-model interactive config builder. Lists `.gguf` files under `ModelsDir` (marks already-configured ones with `*`), prompts for every model param (`CtxSize`, `GpuLayers`, `Parallel`, `BatchSize`, `UbatchSize`, `CacheTypeK/V`, `FlashAttn`, `Jinja`, `NCpuMoe`, sampling, `ChatTemplateKwargs`), writes `models\<id>.psd1`, and updates `server.psd1` with `ActiveModel = '<id>'`. Re-runnable via "Configure model" Start Menu shortcut to switch active model or refresh params.
- `llama-cpp.nsi.template` — NSIS installer template with placeholders (`@VERSION@`, `@STAGING_DIR@`, `@WEBUI_DIR@`, `@OUTPUT_FILE@`) replaced by `03-package.ps1`. Has two custom nsDialogs pages (server config: Port / Bind; webui config: Port / Bind, conditional on SecWebUI) that drive `config-server.ps1 -NonInteractive` and `config-webui.ps1 -NonInteractive` post-install to write the PSD1 files. Model-dependent params (CtxSize, GpuLayers, Parallel, CacheTypeK/V, FlashAttn, sampling, etc.) are NOT prompted at install — created lazily per-model by config-model.ps1 on first run. Start Menu shortcuts target `pwsh.exe` (PowerShell 7+).
- `relocate-venv.py` — bundled into `build\webui-venv\` by `03-package.ps1` so NSIS can run it post-install to rewrite Scripts\*.exe shebangs and `pyvenv.cfg` from the build path to the install path.
- `.webui_secret_key` — generated by Open WebUI at first run for JWT signing. Gitignored.

## Key conventions

- `common.ps1` is dot-sourced by all scripts that need `$cfg` or ROCm PATH (everything except `00-install-prerequisites.ps1`, `01-configure.ps1`, and the `resources/` runtime scripts). Build/package scripts additionally call `Enable-VsDevShell`.
- The Threads default (`config-server.ps1`) and the Threads fallback (`resources\run-model.ps1`) use the **same formula**: `cores - 2` if `> 8`, otherwise `max(1, cores - 1)`. Don't let them drift.
- `04-run.ps1` does **not** duplicate the llama-server arg-building logic — it invokes `resources\run-model.ps1` directly with `-ServerExe` overridden. Any new server flag is added once, in the resources script.
- Config values use single-quoted strings in `.psd1` (no backslash escaping needed).
- All generated PSD1 files are written with `-Encoding utf8NoBOM` (NSIS reads them through PowerShell's `Import-PowerShellDataFile`, which is BOM-sensitive on some Windows builds).
- All build artifacts (source clones, cmake binary dir, webui venv, NSIS staging) live under `.\build\` (gitignored). The final installer goes to `.\dist\` and the sccache cache to `.\.sccache\` — both at the repo root so they survive `rm -rf build\`.
- The installer registers in `HKLM\Software\llama.cpp` (InstallDir, WebUIDir) and supports upgrades (uninstalls previous version first). User runtime state lives under `%LOCALAPPDATA%\llama.cpp\`: `config\server.psd1` (llama-server runtime + ModelsDir + ActiveModel pointer), `config\webui.psd1` (Open WebUI bind/port), `config\models\<id>.psd1` (per-model: model path, Jinja, NCpuMoe, sampling), `data\` (Open WebUI's `webui.db` / `vector_db/` / `uploads/`), `logs\` (`llama-server.log`, `open-webui.log`). The uninstaller asks (defaulting to No) whether to delete the whole tree, so it survives reinstall/upgrade by default.
- sccache is auto-detected; cache stored in `.sccache/` (gitignored). Uses `SCCACHE_MAX_FRAME_LENGTH=104857600` for large GPU objects.
- Open WebUI dependencies are installed with relaxed pins (`==` → `>=`) to handle yanked packages on PyPI, then the package itself with `--no-deps`.
- Frontend build is tracked via `.build-commit` marker file in the Open WebUI repo root to skip unnecessary rebuilds.
